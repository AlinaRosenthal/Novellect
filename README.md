# Novellect

Локальная библиотека книг с гибридным поиском, двумя режимами работы и локальной LLM без API.

## Что делает проект

Novellect индексирует книги (`.txt`, `.fb2`, `.pdf`, `.epub`, `.zip`), строит гибридный индекс (BM25 + эмбеддинги + rerank), а затем отвечает на запросы пользователя двумя способами:

- **Lite / слабое железо** — текущий стабильный агентный пайплайн без локальной генерации LLM.
- **Full / LLM + агент** — поверх того же поиска добавляется локальная LLM, которая строит план поиска, запускает несколько подзапросов и формирует grounded-ответ по найденным фрагментам.

Оба режима используют одну и ту же библиотеку и один и тот же индекс, поэтому базовое качество поиска не теряется.

## Быстрый старт

```bash
pip install -r requirements.txt
streamlit run app.py
```

## Режимы работы

### 1) Lite / слабое железо

Используется текущий агентный пайплайн:

- анализ типа запроса;
- безопасная очистка формулировки;
- гибридный поиск по локальному индексу;
- детерминированная выдача без генеративной модели.

Этот режим нужен для устройств с ограниченными ресурсами: CPU-only, мини-ПК, ноутбуков без GPU, Raspberry-подобных платформ.

### 2) Full / LLM + агент

Добавляется локальная LLM без API:

- LLM строит план поиска;
- агент делает несколько поисковых проходов по локальной библиотеке;
- ответ синтезируется только по найденным источникам;
- в UI показываются источники, подзапросы и трассировка агента.

По умолчанию в конфиге используется `Qwen/Qwen2.5-0.5B-Instruct`, но можно указать любой другой `model id` или локальный путь.

## Архитектура

### Индексация

- конвертация книг в текст;
- chunking по абзацам;
- вычисление эмбеддингов;
- сохранение `storage.json` + `vector_db.npz`;
- извлечение текстовых признаков книги для рекомендательных запросов.

### Поиск

- BM25 по чанкам;
- book-level aggregation;
- title lookup;
- опциональный reranker;
- гибридная выдача с дедупликацией книг.

### Агентный слой

- **Lite**: `QueryAnalyzerAgent -> RetrievalAgent -> RankingAgent -> ResponseAgent`
- **Full**: `QueryAnalyzerAgent -> LLMPlannerAgent -> SearchToolAgent -> GroundedAnswerAgent`

## Локальная LLM без API

В проекте нет зависимости от внешних API. Полная версия работает через локальную модель на базе `transformers`.

Поведение сделано безопасным:

- если LLM не загрузилась, проект не ломается;
- full-режим автоматически откатывается к fallback-планированию и grounded-ответу;
- lite-режим всегда остаётся доступным.

## Осторожное дообучение и LoRA

В sidebar добавлен отдельный блок **«Дообучение embedding-модели»**.

Что делает адаптация:
- поддерживает два режима: полный fine-tuning и LoRA;

- строит синтетические query-document пары по уже проиндексированной библиотеке;
- запускает лёгкую domain adaptation для `paraphrase-multilingual-MiniLM-L12-v2`;
- сохраняет адаптированную модель в `fine_tuned_literary_model`;
- новая embedding-модель начинает использоваться только после успешного завершения адаптации.

Это сделано отдельно от основного режима, чтобы не ухудшать базовое качество проекта.

## UI

В интерфейс добавлены:

- переключатель между `Lite` и `Full`;
- настройка устройств для эмбеддингов, reranker и LLM;
- настройка `LLM model id`, `max_new_tokens`, `temperature`;
- анализ вычислительных требований;
- блок адаптации embedding-модели с выбором Full fine-tuning / LoRA;
- трассировка full-агента с подзапросами и источниками.

Вкладка **«Что улучшено»** удалена.

## Анализ аналогов и ЦА

### Целевая аудитория

- студенты и преподаватели, которым нужен локальный демонстрационный RAG-проект;
- пользователи домашних библиотек и цифровых архивов;
- команды, которым нужен офлайн-поиск по коллекции текстов без внешнего API.

### Ближайшие аналоги

- обычный полнотекстовый поиск по файлам;
- векторные RAG-демо без режима слабого железа;
- книжные рекомендательные системы без прозрачной grounded-выдачи.

### Чем проект отличается

- есть **два режима** под разные вычислительные бюджеты;
- full-режим использует **локальную LLM + агента**, а не только поиск;
- результаты в full-режиме привязаны к источникам;
- есть **контролируемая адаптация embedding-модели (включая LoRA)**, а не только базовая модель.

## Вычислительные требования

### Lite

- ориентир: **2–4 ГБ RAM**;
- GPU не требуется;
- подходит для слабого железа и CPU-only.

### Full

- ориентир: **4–10 ГБ RAM** для модели уровня `0.5B`;
- желательно **3+ ГБ VRAM** для комфортной работы, но CPU fallback поддержан;
- первый full-запрос может занять больше времени из-за загрузки локальной модели.

## Тестирование

В проект добавлен smoke test, который проверяет:

- индексацию книг;
- поиск в `Lite`;
- поиск в `Full`;
- работу full-режима через встроенную `stub-LLM`, чтобы тест не зависел от внешней загрузки модели.

Запуск:

```bash
python tests/smoke_test.py
```

Также можно проверить синтаксис проекта:

```bash
python -m compileall .
```

## Структура новых возможностей

- `llm_runtime.py` — загрузка и вызов локальной LLM, fallback и stub-режим для тестов.
- `fine_tuning.py` — безопасная адаптация embedding-модели (включая LoRA).
- `agents.py` — разделение на `Lite` и `Full` оркестраторы.
- `runtime_config.py` — конфигурация режимов и оценка требований.
- `app.py` — UI-переключение режимов и управление адаптацией.
